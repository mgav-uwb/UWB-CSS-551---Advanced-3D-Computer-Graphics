Lab 01 — Unity setup & the interactive loop

Week 1 · Thursday in-person studio · Dr. Marcel Gavriliu · bring your laptop

Tuesday's lecture built the mental model: interactive graphics is a loop (input → update → redraw) organized as MVC. Today you install the workbench and watch that loop run in real code — the same two projects we toured on slides. Work at your own pace; the instructor is circulating.

Goal for today: Unity installed and running, the tour projects open, and you have changed a variable, predicted the result, and confirmed it. That is the whole "Practice" half of Code–Demo–Practice.

1 · Install Unity 6 (Unity Hub)

We standardize on one editor version so everyone's projects open identically. Install Unity Hub first, then use it to install the pinned editor.

  • Download Unity Hub from unity.com/download and install it.
  • Sign in / create a free Unity account (a personal license is fine for coursework).
  • In Hub → Installs → Install Editor, pick Unity 6000.3.11f1 LTS (the pinned version for this course).
  • When prompted for modules, the default set is fine; you do not need mobile/console build support.
  • Let it finish (this download is large — start it first thing).
Version matters. Opening a project in a different editor version can silently upgrade it and cause diffs that are not your work. If Hub offers to open a project in a newer version, decline and add 6000.3.11f1 instead.

3 · Run the loop — 1.1.IntroToTool

Open the scene, press Play (the ▶ button at the top), and watch. Then read the two scripts under Assets/Source/: MoveInX.cs and BounceUpAndDown.cs.

POKE 1

Change the drive speed

In MoveInX.cs, the Update() body advances x every frame:

p.x += 1.0f * Time.smoothDeltaTime;
Predict first: if you change 1.0f to 3.0f, what changes about the motion — the direction, the speed, or the starting point? Write your guess down.

Do it: change the value, save the file, return to Unity (it recompiles), press Play. Compare to your guess.

POKE 2

Change the bounce range

BounceUpAndDown.cs has a field yRange that bounds the vertical bounce:

public float yRange = 10f;
Predict first: set yRange to 3f. Will the object bounce faster, or over a shorter distance, or both? Why?

Do it: change it, Play, and watch. Then ask yourself: yRange is a public field — where else could you have changed it without editing the file? (Hint: the Inspector.)

Frame-rate note. Notice MoveInX multiplies by Time.smoothDeltaTime. Delete the * Time.smoothDeltaTime so it reads p.x += 1.0f;, Play, and watch the speed jump. That is a per-frame step instead of a per-second step — put it back when you have seen it.

4 · Read the Controller — 1.2.SimpleGUI

Open 1.2.SimpleGUI. Press Play: use the buttons to create cubes and the toggle to show/hide the sphere. Then open MainController.cs (under Assets/Source/UI Support/) and read it top to bottom — it is the whole app.

POKE 3

Change where new cubes appear

In MainController.cs, CreateNewCube() places each cube at:

cube.transform.position = new Vector3(10, 0, 10);
Predict first: change it to new Vector3(0, 5, 0). Where will the next created cube appear relative to the old spot?

Do it: change it, Play, click Create, and confirm. Then find the three AddListener(...) calls in Start() — each is one button/toggle wired to one handler. Which handler runs when you click Create?

5 · SliderWithEcho — map the MVC triad

Open 2.3.SliderWithEcho (the project we toured in lecture Part 4). Press Play: click to place a sphere, drag the slider, and watch its radius change while the number echoes. Then open the three source files under Assets/Source/ and match each to a box from Tuesday.

Trace it: follow one slider drag from SliderValueChange → RadiusChanged → TheWorld.SetSelelectedRadius → the sphere redrawing. Notice every arrow points one way: Controller → Model → View. That one-way flow is the whole point of MVC.

6 · MP1 kickoff

MP1 (Machine Problem 1) — orientation goes out this week. It is your first implement-and-replace: you will orient an object using only primitive vector/matrix operations (Vector3.Dot, Vector3.Cross, Vector3.Normalize, component access) and then check your result against what the engine's own helper produces. The composited helpers (Transform.LookAt, Quaternion.LookRotation) are off-limits where the assignment says so — building them is the assignment.

Details, due date, and points are on Canvas (the single source of truth). Skim the MP1 write-up before you leave today and post questions on the Canvas discussion board.
