Week 7 · Thursday in-person studio · Dr. Marcel Gavriliu · bring your laptop and a half-formed project idea

Tuesday's lecture built a triangle mesh by hand — a vertex array plus an index array of triangle triples, with normals computed as averaged cross products. Today has two halves. First, you run Sung's real MyMesh projects: read the hand-built arrays, turn on averaged normals, and drag a vertex to see geometry and normals move together. Second, the Final Project proposal workshop — brainstorm, pitch, peer-review, and revise a one-paragraph proposal.

Predict before you run — every time. In the studio half, say (or jot) what you expect the code to do before you press Play. The point is to catch the moment your mental model and the screen disagree.

Part A · MyMesh studio ~50 min

0 · Get the Topic7 projects

The projects are in the course materials on Canvas (Topic7-PolygonalModeling). Download, unzip somewhere without spaces in the path, and add each folder in Unity Hub (Projects → Add → Add project from disk). Open with the pinned editor from Lab 01. Today's projects:

  • 7.3.Simple2x2Mesh, 7.5.NormalAverageSupport, and 7.4.SimpleAttemptAtMaipulation each open and compile without errors.
  • In each, find MyMesh.cs (and the partial-class files MyMesh_NormalSupport.cs / MyMesh_Manipulate.cs where present) under Assets/.

1 · Read the hand-built mesh

Goal. Confirm the 9 vertices and 8 triangles on paper match the code. Open 7.3.Simple2x2Mesh/Assets/MyMesh.cs and read the v[0..8] assignments and the t[0..23] triples.

CHECKPOINT 1a

Match the vertices to the diagram

On paper, draw the 3×3 lattice from lecture (row 0 at the back, row 2 at the front) and label each corner with its v[i] position from the code.

Predict first: the nine positions are three rows of three, row = Z, column = X, spanning −1 to 1. Predict v[4]'s position before you read it (hint: it is the center). Confirm, then note that our lecture demo used the same layout but spanned ±1.5.
CHECKPOINT 1b

Sung's split vs. the demo's split

Read the eight t[] triples. Sung fans each quad from its lower-left corner:

t[0..5] = 0,3,4,  0,4,1     // quad (0,0): fan from v0
// ...the lecture DEMO split the same quad the other way:
//   tri0 = 0,3,1     tri1 = 1,3,4     // split along the v3-v1 diagonal
Predict first: both are valid meshes of the same 9 vertices — same surface, different diagonal and index order. Predict: does the rendered floor look any different between Sung's fan and the demo's split? (It should not — same triangles cover the same area.) Confirm the mesh renders as a flat floor, and note where the two splits' diagonals differ.

2 · Turn on averaged normals

Goal. See the per-vertex normal averaging from lecture, drawn. Open 7.5.NormalAverageSupport and read MyMesh_NormalSupport.cs: FaceNormal (normalized cross of two edges) and ComputeNormals (each n[i] = normalized sum of its incident face normals).

Where to look: ComputeNormals builds eight triNormal[] face normals, then sets each vertex normal to a sum. The center n[4] sums six faces; a corner like n[6] sums one; an edge like n[1] sums three — exactly the "how many triangles touch this vertex" count from lecture.
CHECKPOINT 2a

Predict the flat-mesh normals

Run the project with the mesh flat (no vertex lifted). The normals are drawn as line segments at each vertex.

Predict first: every face of a flat floor has normal (0,1,0), so every averaged vertex normal is straight up. Predict what the normal segments look like (all parallel, pointing up?). Run and confirm.
CHECKPOINT 2b

Count the faces at each vertex

On paper, for the center vertex, a mid-edge vertex, and a corner vertex, list which triangles touch it.

Predict first: center touches 6, mid-edge 3, corner 1 (or 2, depending on the split). Predict each count, then check it against the sum in ComputeNormals for that n[i]. This is why n[4] has six terms.

3 · Drag a vertex

Goal. See geometry and normals move together. Open 7.4.SimpleAttemptAtMaipulation; each vertex has a draggable sphere handle (InitControllers), and the Update loop reads the handles back into the vertex array each frame.

CHECKPOINT 3a

Lift the center vertex

Grab the center vertex's sphere and drag it up. Watch the surface and (in 7.5) the normals.

Predict first: lifting the center tents the surface; the six faces around it tilt, so their normals swing away from vertical, and the averaged vertex normals follow — but the peak's own normal, by symmetry, stays straight up. Predict which normals visibly swing and which stay put, then drag and confirm. This is the lecture demo's lift, in the real project.
CHECKPOINT 3b

Why recompute at all?

Find the line in Update that calls ComputeNormals after re-reading the vertices.

Predict first: positions and normals are coupled — move a vertex and the old normals are stale. Predict what the lift would look like if ComputeNormals were not called each frame (hint: the shading would not match the new shape). State it in one sentence.

Part B · Final Project proposal workshop ~50 min

The Final Project is your own graphics project, built on the course's techniques — meshes, the camera, transforms, and (soon) texturing and lighting. It starts with a one-paragraph proposal. Today's workshop turns a rough idea into a peer-reviewed pitch you can submit.

The proposal template and requirements are on Canvas (the single source of truth) — due dates, points, and required sections live there, not here.

ROUND 1 · BRAINSTORM ~8 min

List ideas, no filtering

Spend the first few minutes dumping candidate ideas — a small interactive scene, a modeling tool, a rendering effect, a game mechanic. Do not judge them yet; quantity first. Then pick one to develop.

Scope check: pick something you can actually finish in the weeks remaining. A finished small thing outscores an unfinished big one. Good seeds reuse what you have built: a mesh you can edit, a camera you can fly, a scene graph you can animate.
ROUND 2 · PITCH ~15 min

Write the one-paragraph pitch

Using the Canvas template, write a single paragraph stating three things:

  • the idea — what you are building, in one or two sentences a classmate can picture
  • the core technique — the main graphics concept it exercises (mesh construction, camera control, hierarchical animation, ...)
  • the minimal deliverable — the smallest version you are confident you can finish and demo
ROUND 3 · PEER FEEDBACK ~18 min

Swap pitches, ask the two questions

Pair up and read each other's pitch. As the reviewer, ask the two questions that matter most:

  • Is the scope finishable? — can this be built and demoed in the time left? If not, what is the smaller version?
  • What is the single biggest risk? — the one thing most likely to sink it (an unfamiliar technique, an asset you do not have, a hard algorithm)?

Give each other concrete, specific feedback — "narrow it to X" beats "sounds good". Then switch roles.

ROUND 4 · REVISE ~9 min

Tighten and submit

Revise your pitch using the feedback: narrow the scope if it was too big, name the risk and how you will handle it, and make the minimal deliverable concrete. Then submit on Canvas per the template.

Pull the instructor over with a specific question — "is this scope realistic?" or "which technique does this need?" This is the best help-per-minute of the week.
