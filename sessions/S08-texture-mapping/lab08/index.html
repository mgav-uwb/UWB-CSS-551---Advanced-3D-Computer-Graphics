Tuesday's lecture turned a bare mesh into a textured one: a UV coordinate per vertex, a 3×3 matrix that places the image, a sampler that reads a color back, and a procedural texture that is pure math. Today has two halves. First, you run Sung's real Topic8 texture projects: read the UVs, place the image by hand, blend two textures, and inspect the procedural checker. Second, dedicated MP6 lab time — its texture-placement part is exactly Tuesday's Part 3, on your own mesh.

Predict before you run — every time. Each placement poke below asks you to state what the offset / rotation / tile will do before you press Play. The point is to catch the moment your mental model and the screen disagree — that gap is where the learning is.

Part A · Texture studio ~55 min

0 · Get the Topic8 projects

The projects are in the course materials on Canvas (Topic8-TextureMapping). Download, unzip somewhere without spaces in the path, and add each folder in Unity Hub (Projects → Add → Add project from disk). Open with the pinned editor from Lab 01. Today's projects:

  • 8.1.UVOnVertices, 8.2.OurOwnTexturePlacement, 8.6.MultiTexturing, and 8.7.SynthesizedTextures each open and compile without errors.
  • In each, find MyMesh.cs (and, where present, TexturePlacement.cs / CheckerTexture.cs and the .shader file) under Assets/.

1 · Read the per-vertex UVs

Goal. Confirm the 9 UVs on paper match the code. Open 8.1.UVOnVertices/Assets/MyMesh.cs and read the uv[0..8] assignments beside the v[0..8] positions.

CHECKPOINT 1a

Match the UVs to the grid

On paper, draw the 3×3 lattice from lecture and label each corner with its uv[i].

Predict first: the mapping is (u,v) = (col/2, row/2), so the four grid corners take the four corners of the unit square and the center vertex v4 takes (0.5, 0.5). Predict uv[4] and uv[8] before you read them, then confirm against the array.
CHECKPOINT 1b

Interpolation between corners

Only the corners carry UVs; the rasterizer interpolates the rest.

Predict first: the spatial midpoint of edge (0,3) — halfway from v0 to v3 — samples the texture at which (u,v)? (Average the two endpoint UVs.) Predict it, then reason about what that point of the image looks like on the rendered floor.

2 · Place the image by hand

Goal. See offset and tiling as a UV transform. Open 8.2.OurOwnTexturePlacement and read TexturePlacement.cs: each frame it multiplies the saved UVs by Scale and adds Offset. Select the mesh in the Inspector to expose those two fields.

Where to look: TexturePlacement.cs saves the authored UVs in mInitUV, then every Update() sets uv[i] = Offset + mInitUV[i] * Scale. Scale is the tile factor; Offset slides the image. (This version scales about the corner, no rotation — the lecture's uvMat3 adds the center sandwich and a rotation.)

POKE 2a · OFFSET

Slide the image

Set Offset to (0.25, 0) and run.

Predict first: adding 0.25 to every u shifts the image by a quarter of one copy along U. Predict which way the picture slides and by how much, then run and confirm. (Which lecture matrix cell does this offset live in?)
POKE 2b · TILE

Repeat the image

Set Scale to (2, 2), then (6, 6). Make sure the texture's wrap mode is Repeat (Inspector → the texture asset).

Predict first: scaling the UVs by 2 makes the image fit twice on each axis (repeat wrap turns the extra range into copies); by 6, six times. Predict how many copies you will see at Scale = (6,6), and predict what happens if the wrap mode is Clamp instead. Run both and confirm.
POKE 2c · ROTATE (on our demo)

Spin about the center

TexturePlacement.cs has no rotation, so use the lecture demo (the uv-placement sandbox on the course hub, with the rot slider). Set tile = 2, rot = 45°, offU = 0.25.

Predict first: the placement matrix should read [1.41, 1.41, -0.66 / -1.41, 1.41, 0.50 / 0.00, 0.00, 1.00] (as the panel displays it, from lecture). Predict the diagonal value from 2 · cos45° before you look, then read the panel and check every cell. Does the checker spin about its center or a corner?

3 · Blend two textures

Goal. See multi-texturing. Open 8.6.MultiTexturing and read the fragment shader (Assets/Materals/451NoCullShader.shader): it samples _MainTex and _SecTex and blends them.

POKE 3a

Change the blend weights

Find the line col = 0.2 * col + 0.8 * c2; in frag.

Predict first: the fragment is 20% the first texture, 80% the second. Predict what the surface looks like if you swap the weights to 0.8 / 0.2, then edit, save, and confirm. What blend weights would show only the second texture?

4 · Inspect the procedural checker

Goal. See a texture that is pure math. Open 8.7.SynthesizedTextures; read CheckerTexture.cs (it pushes URepeat, VRepeat, Color1, Color2 to the material) and the checker block in the shader.

POKE 4a

Predict the checker from the math

Read the shader lines that compute scaledU, scaledV, and the parity test if (uEven != vEven).

Predict first: the color is picked by the parity of floor(u · URepeat) + floor(v · VRepeat). Predict how many checker squares appear at URepeat = VRepeat = 4, then set them and confirm. No image is loaded — where do the colors come from?

Part B · MP6 lab time ~45 min

MP6 (the polygonal-modeling editor, out last week) has a texture-placement part — exactly Tuesday's Part 3, applied to your own mesh. Use this block to get it working with the instructor in the room.

Requirements, due dates, and points are on Canvas (the single source of truth). Nothing new is due today and nothing new goes out — this is work time on the assignment you already have.

FOCUS

Get placement working on your mesh

  • Assign a sensible UV per vertex (the (col/2, row/2) pattern generalizes to your n×n grid).
  • Wire up offset / rotation / tile controls that build a placement transform — the same three knobs as the lecture demo.
  • Verify with a known case: at tile = 2, offset 0, rotation 0, your texture should repeat exactly twice per axis (repeat wrap). If it does not, check the wrap mode and the scale.
Pull the instructor over with a specific question — "my tiling repeats the wrong number of times" or "my rotation spins about the corner, not the center." A specific question is the best help-per-minute of the week.
STRETCH

If placement already works

Try: make your rotation spin about the texture center (the T(c) · R · T(-c) sandwich), not the corner. Predict the difference on screen before you change the code, then confirm the image spins in place rather than swinging away.

Before you leave

  • You matched 8.1's 9 UVs to the grid and predicted the edge-midpoint interpolation.
  • You slid, tiled, and (on the demo) rotated the texture in 8.2 / the sandbox, predicting each result first.
  • You changed the blend weights in 8.6 and inspected the procedural checker math in 8.7.
  • You spent MP6 lab time getting texture placement working on your own mesh, and verified the tile-2 case.
