3 · Run the loop — 1.1.IntroToTool

Read this first — the loop. A console program runs top to bottom and exits. An interactive graphics program never returns: a framework runs this forever, and your code hooks the middle:

initialize state                     ← your Start()
repeat forever:
    read input     (mouse, keys)     ┐
    update state   (move things)     ├ ← your Update(), once per frame
    redraw         (state → pixels)  ┘

At 60 frames per second each trip gets 16.7 ms — everything must fit or the app stutters. The screen is never the truth: it is a rendering of the state, only as fresh as the last redraw. And a frame is not a fixed slice of time — that is why per-frame motion multiplies by Time.smoothDeltaTime (seconds since last frame): rate × seconds = this frame's step, same speed on every machine. You will see both in the code below.

Unity in four nouns. A Scene is the current collection of GameObjects (a cube, a camera, a light). A GameObject is a bag of Components — capabilities bolted on (every one has a Transform: position/rotation/scale). A Script is a Component you write in C#; Unity calls its Start() once and its Update() every frame. Two gotchas worth naming now: a public field shows up in the Inspector and gets its value by dragging a scene object onto it; and transform.localPosition is position relative to the parent (transform.position is world) — the difference bites once objects nest. One more: transform.localPosition is a C# struct (a value copy) — read it into a local, modify, and write it back, or nothing moves. You WILL hit this bug; now you know its name.

Open the scene, press Play (the ▶ button at the top), and watch. Then read the two scripts under Assets/Source/: MoveInX.cs and BounceUpAndDown.cs.

POKE 1

Change the drive speed

In MoveInX.cs, the Update() body advances x every frame:

p.x += 1.0f * Time.smoothDeltaTime;
Predict first: if you change 1.0f to 3.0f, what changes about the motion — the direction, the speed, or the starting point? Write your guess down.

Do it: change the value, save the file, return to Unity (it recompiles), press Play. Compare to your guess.

POKE 2

Change the bounce range

BounceUpAndDown.cs has a field yRange that bounds the vertical bounce:

public float yRange = 10f;
Predict first: set yRange to 3f. Will the object bounce faster, or over a shorter distance, or both? Why?

Do it: change it, Play, and watch. Then ask yourself: yRange is a public field — where else could you have changed it without editing the file? (Hint: the Inspector.)

Frame-rate note. Notice MoveInX multiplies by Time.smoothDeltaTime. Delete the * Time.smoothDeltaTime so it reads p.x += 1.0f;, Play, and watch the speed jump. That is a per-frame step instead of a per-second step — put it back when you have seen it.

Read this first — MVC. An interactive app juggles three concerns: what exists (the Model — the scene state, the single source of truth), what it looks like (the View — this frame's rendering of the model), and what the user did (the Controller — input mapped to model changes). The rule that makes it work: data flows ONE way — Controller writes the Model, Views only read it. Two views of one model always agree; a view that could write back would create two truths.

See it live (2 min): open the “one model, two views” demo — three sliders (Controller) drive three numbers (Model); a 3D cube and a matrix panel (two Views) both re-render from those numbers. Drag a slider and watch both views agree, always.

Change where new cubes appear

In MainController.cs, CreateNewCube() places each cube at:

cube.transform.position = new Vector3(10, 0, 10);
Predict first: change it to new Vector3(0, 5, 0). Where will the next created cube appear relative to the old spot?

6 · The course rule, and MP1

Implement-and-replace. Unity ships helpers that do the interesting math for you (Transform.LookAt, Quaternion.LookRotation, …). Where the assignment says so, those are off-limits: you may use only primitive operations — and you build the helper yourself, then check your build against the engine's version (the engine is your answer key). The point of the course is to know what the helper does, not to call it.
