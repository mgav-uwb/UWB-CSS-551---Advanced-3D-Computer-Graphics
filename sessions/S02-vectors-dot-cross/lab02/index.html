Lab 02 — Vectors, planes & the in-front-of test

Week 2 · Thursday in-person studio · Dr. Marcel Gavriliu · bring your laptop

Tuesday's lecture built the toolbox: a vector is a displacement, the dot product measures alignment (angle, projection, which side), and the cross product makes perpendicularity (a normal, an area, a frame). Today you turn that math into code in three exercises — EX1 (vectors), EX2 (plane), and EX3 (in front of). Work at your own pace; the instructor is circulating.

The course rule is in force where each exercise says so. Build from the primitive operations only: Vector3.Dot, Vector3.Cross, .normalized, .magnitude, component access, and vector + - *. Do not reach for Transform.LookAt, Vector3.Angle, Vector3.Project, or any "do it for me" helper — building them is the exercise.

1 · EX1 — Vectors: march, aim, and fire

Goal. Make an object march along a line, draw the two guide lines, and — when you press Space — launch a ball from the marcher straight at a moving target. This is the aim-line / traveling-ball pattern from lecture Part 1 and Part 5.

Allowed API: a - b (displacement), v.normalized, v.magnitude, v * scalar, Time.deltaTime, transform.localPosition, Input.GetKeyDown. Not Vector3.MoveTowards or Vector3.Lerp — do the march yourself.
CHECKPOINT 1a

March ObjA from p1 toward p2

Each frame, move ObjA a step of ObjASpeed * Time.deltaTime along the direction from p1 to p2; when it reaches p2, restart at p1.

Predict first: the step is speed * dt * (p2 - p1).normalized. If you forget .normalized and use the raw (p2 - p1), does ObjA move faster, slower, or the same — and does the answer depend on how far apart p1 and p2 are? Write your guess, then try it.
CHECKPOINT 1b

Aim and fire the projectile

Inside the if (Input.GetKeyDown(KeyCode.Space)) block, start the ball at ObjA's position and give it an aim direction toward Red. Each frame while it is active, march it at ProjectileSpeed; hide it when it reaches Red.

Predict first: you compute aim = (Red.position - ObjA.position).normalized once, at fire time. Red keeps drifting after you fire. Will the ball still hit Red, or sail past where Red was? What would you change to make it track?
CHECKPOINT 1c

Implement SetLine(p1, p2, cylinder)

Draw a cylinder stretched from p1 to p2. You need its midpoint (a position), its direction (a unit displacement), and its length (a magnitude) to set the transform's position, up vector, and y-scale.

Predict first: which of the three — midpoint, direction, length — comes from a subtraction, which from .normalized, and which from .magnitude? Match them before you write the function.

2 · EX2 — Plane: draw it from a normal and D

Goal. Display the white plane from its normal and D, and draw a cylinder standing along the plane's normal so you can see which way it faces. The plane convention (stated in the file) is n · P = D, i.e. n · P − D = 0.

Allowed API: Vector3.Dot, v.normalized, transform.up / transform.rotation, Quaternion.FromToRotation (to point the plane's up along the normal), component access. Reuse your SetLine from EX1 to draw the normal.
CHECKPOINT 2a

Orient the plane to its normal

Rotate WhitePlane so its up axis points along WhitePlaneNormal, and offset it so it sits the correct distance out (D along the unit normal).

Predict first: with WhitePlaneNormal = (0, 1, 0) and D = 0, where does the plane sit? Now predict where it moves when you set D = 3 with the same normal. Then set the normal to (1, 2, 2) — before running, normalize it on paper (|n| = 3) and say which way it tilts.
CHECKPOINT 2b

Draw the normal as a cylinder

Use SetLine to draw a short cylinder from the plane's center along its unit normal, so the plane's facing direction is visible.

Predict first: if you draw the cylinder along WhitePlaneNormal without normalizing first, and the normal is (1, 2, 2), will the cylinder be the right length, three times too long, or point the wrong way? Why?

3 · EX3 — In front of: the which-side test

Goal. Given the white plane and a movable point P, color P green when it is in front of the plane (the normal's side) and red when it is behind. This is the sign of n · P − D from lecture Part 4.

Allowed API: Vector3.Dot, WhitePlane.up (the plane's normal), P.localPosition, component access, and the color setter shown in the file (GetComponent<Renderer>().material.color). One dot, one subtraction, one comparison — no more.
CHECKPOINT 3a

Compute the signed side

Let n = WhitePlane.up. Build D from a known point on the plane (its position), then test the sign of Vector3.Dot(n, P.localPosition) - D and set the color.

Predict first: before you move anything, predict the color for a point clearly above the plane, then for one clearly below. Which sign is "front"? Drag P slowly through the plane and watch the exact moment it flips.
EX3 gotcha (from the file's own comment): move the plane so its localPosition x, y, z are not all zero. If the plane sits at the origin, D = 0 and the test can look right for the wrong reason — you would be testing n · P alone. A non-zero plane position is what actually exercises the − D term.
