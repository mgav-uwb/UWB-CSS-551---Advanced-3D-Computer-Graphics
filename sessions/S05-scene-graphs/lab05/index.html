1 · Trace CompositeXform

Goal. See the recursion compute every node's world matrix in one top-down pass. Open SceneNode.cs and re-read the one line that is the whole rule:

mCombinedParentXform = parentXform * orgT * trs;   // W = W_parent · L
foreach (SceneNode child in ChildrenList)
    child.CompositeXform(ref mCombinedParentXform); // pass THIS node's world down
Where to look: 5.1.SceneNode+PrimitiveList/Assets/Source/Model/SceneNode.cs — the CompositeXform(ref Matrix4x4 parentXform) method. The root call passes Matrix4x4.identity as the top parent (find where TheWorld kicks it off).
CHECKPOINT 1a

Predict the call stack

The tree is base → arm → hand. Trace the recursion by hand on paper.

Predict first: write the three assignments the recursion makes, in order — W_base = ?, W_arm = ?, W_hand = ? — each as a product of local matrices. Which node's CompositeXform receives identity? Which receives W_arm? Confirm your W_hand is the product of all three locals, root-most on the left.
CHECKPOINT 1b

Where does geometry get its matrix?

Read NodePrimitive.LoadShaderMatrix: m = nodeMatrix * p * trs * invp.

Predict first: nodeMatrix is the node's world matrix handed in by CompositeXform. Predict what p * trs * invp is (hint: it is a pivot idiom from Lab 04). If a node carried two primitives with different Pivot values, would they share a world frame? Explain in one sentence, then confirm by adding a second primitive to a node's PrimitiveList.

2 · Add a node, predict its composite

Goal. Extend the tree and predict the new node's world matrix before running. In the Hierarchy, duplicate the hand node (or create a new SceneNode) and wire it as a child of the arm, alongside the existing hand.

How to wire it: select the arm node; in its SceneNode component, add your new node to the ChildrenList. Give the new node a primitive (add a NodePrimitive to its PrimitiveList) and a clear local offset in its Transform (say localPosition = (0, 1, 0)).
CHECKPOINT 2a

Predict the world position

Your new node is a child of the arm with local offset (0, 1, 0).

Predict first: its world matrix is W_new = W_arm · L_new. Without computing every entry, predict where its origin lands relative to the existing hand — same tip, higher, offset? Then bend the arm (armBend-style rotation on the arm node) and predict: does your new node swing with the arm, or stay behind? Run and check the box moved as one with the arm subtree.
CHECKPOINT 2b

Move the parent, watch the children

Select the base and yaw it; then select the arm and yaw it.

Predict first: yawing the base should sweep base, arm, hand, and your new node together; yawing the arm should move arm, hand, and your new node but leave the base fixed. Predict which nodes move for each edit before dragging, then confirm. Name the invariant this demonstrates in your own words.

3 · Re-pivot a joint

Goal. Change where a node hinges and predict the swing. In lecture, the arm bent about its lower end (the joint) because of the pivot sandwich T(p)·R·T(-p). Now move that pivot.

Where to look: in 5.2.SceneNodeControl, select the arm node; find its NodeOrigin (the orgT offset in CompositeXform) and/or the primitive's Pivot. Change one and observe how the arm rotates.
CHECKPOINT 3a

Predict the fixed point

Set the arm's pivot to its lower end (the joint), then to its center, then to its tip.

Predict first: the pivot point is the fixed point of the rotation — it does not move as the node bends. For each of the three pivots, predict which point on the arm stays put and how the far end swings. Point at the spot on screen, then enable rotation and check. Which pivot keeps the hand glued to the tip the way lecture's demo did?
CHECKPOINT 3b

Does the child care where the parent pivots?

With the hand still on the arm's tip, change the arm's pivot and re-bend.

Predict first: the hand's local transform never mentions the arm's pivot — it only says "sit at the arm's tip". Predict: when you move the arm's pivot and bend, does the hand still ride the tip correctly, or does it detach? Explain using W_hand = W_arm · L_hand — what does the hand inherit, and what does it not?

4 · MP4 kickoff — the four-generation hierarchy

MP4 is a four-generation hierarchy built on the same SceneNode you just traced — one generation deeper than tonight's base-arm-hand arm. The CompositeXform recursion already handles any depth, so a fourth level is one more application of W_child = W_parent · L_child.

Get MP4 open now and make a start: sketch your four-level tree on paper (root → child → grandchild → great-grandchild), decide each node's local transform in its parent's frame, and wire the first two generations as SceneNodes. Pull the instructor over with a specific blocker — this is the best help-per-minute of the week.
